01 — Overview

OpenCheckout 개요#

OpenCheckout은 주소·배송·결제·약관 4개 위젯이 하나의 세션 버스를 공유하는 오픈소스 체크아웃 표면입니다. 키 없이 시작하고, 18줄로 통합이 완료됩니다.

Widget-first 모델이란#

전통적인 결제 SDK는 어셈블리 모델입니다. 개발자가 폼 HTML을 직접 작성하고, PG 스크립트를 삽입하며, 각 위젯 간 상태 동기화를 수동으로 관리합니다. OpenCheckout은 이와 반대입니다. 4개의 Shadow DOM 위젯이 공통 이벤트 버스(EventBus<WidgetBusMap>)를 통해 자동으로 상태를 공유하고, 국가 선택·우편번호 유효성·할부 옵션 같은 UX 로직은 SDK 내부에서 처리합니다.

대상 독자#

이 가이드는 한국 D2C(직접판매) 또는 국제배송 서비스에 체크아웃을 구현하는 프론트엔드·풀스택 개발자를 대상으로 합니다. HTML/JS 기초와 async/await에 익숙하면 충분합니다.

React, Vue 등의 컴포넌트 프레임워크에서도 동일한 Vanilla JS API를 그대로 사용합니다. Custom Element 레지스트리에 한 번만 등록되므로 SSR/hydration 환경에서는 typeof window !== "undefined" 가드 후 호출하세요.

코드 주석

  • 1마운트 타깃 4개를 원하는 위치에 배치합니다. 순서는 자유입니다.
  • 2OpenCheckout.load()는 Promise를 반환합니다. await로 완료를 기다립니다. publishableKey 형식: pk_(test|live)_<shard>_<random>
  • 3widgets()는 세션 버스를 생성합니다. 한 페이지에서 복수 인스턴스를 만들 수 있지만, 위젯을 공유하려면 동일한 widgets 참조를 사용해야 합니다.
  • 4setAmount()와 setOrder()는 renderXxx 이전에 반드시 호출해야 합니다. 순서를 어기면 OpenCheckoutValidationError가 throw됩니다.
  • 54개 renderXxx 호출은 각각 위젯 인스턴스를 반환합니다. 이벤트 구독이 필요하면 반환값에 .on()을 사용합니다.
  • 6requestPayment()는 현재 v1에서 mock redirect입니다. 실제 PG 연동은 v1.0.1에서 제공될 예정입니다. 자세한 내용은 requestPayment 섹션을 참고하세요.
i

호출 순서: load() → widgets() → setAmount() + setOrder() → renderXxx() → requestPayment(). 이 순서를 지키지 않으면 OpenCheckoutValidationError가 throw됩니다.

03 — Initialize

Initialize#

SDK를 사용하려면 load() → widgets() → setAmount() + setOrder() 세 단계를 순서대로 실행해야 합니다.

instance.widgets()#

세션 상태와 이벤트 버스를 공유하는 4개 위젯 팩토리를 생성합니다. 동일한 체크아웃 페이지 내 위젯끼리 상태를 공유하려면 반드시 같은 widgets 인스턴스를 사용해야 합니다.

파라미터

이름타입기본값설명
customerKey* string — 고객 식별자. 비회원은 "ANONYMOUS". 회원은 UUID 또는 [A-Za-z0-9\-_=.@] 2–50자, 특수문자(-_=.@) 최소 1개 포함 필수.
locale "ko" | "en" | "ja" | "zh-CN" "en" 위젯 UI 언어. 주소 폼 레이블, 배송 옵션 표시 등에 적용됩니다.

반환: CheckoutWidgets (동기 반환)

setAmount()#

renderXxx() 이전에 반드시 호출해야 합니다. 변경 시 마운트된 모든 위젯에 amount:change 이벤트가 자동 전파됩니다. PAN 스캐너(assertPanFree)를 통과해야 하며, 카드번호 형식의 숫자열이 포함된 값은 거부됩니다.

파라미터 — Money

이름타입설명
amount.value* number 0 이상의 유한 숫자. 음수 또는 NaN은 오류.
amount.currency* "KRW" | "USD" | "JPY" ISO 4217 통화 코드. 현재 3개 통화 지원.

setOrder()#

주문 정보를 세션에 설정합니다. buyerCountry를 지정하면 주소 위젯의 초기 국가가 해당 값으로 설정됩니다.

파라미터 — OrderInfo

이름타입기본값설명
order.id* string — 비어있지 않은 주문 ID. requestPayment redirect URL에 포함됩니다.
order.name* string — 상품명. 비어있지 않아야 합니다.
order.buyerCountry string — ISO-3166-1 alpha-2 국가 코드. 주소 위젯 초기 국가 및 결제 수단 필터링에 사용됩니다.
!

호출 순서 강제: renderAddress, renderShipping, renderPayment, renderAgreement는 모두 setAmount()가 먼저 호출된 상태여야 합니다. 그렇지 않으면 OpenCheckoutValidationError: setAmount must be called before renderXxx가 throw됩니다.

08 — Payment

requestPayment()#

사전 조건이 모두 충족되면 결제 요청을 초기화합니다. v1에서는 게이트웨이 API를 호출하지 않고 successUrl로 mock redirect를 수행합니다.

!

v1 mock redirect: 현재 requestPayment()는 실제 PG 연동 없이 successUrl?paymentKey=mock_<orderId>_<timestamp>&orderId=<id>&amount=<value> 형식으로 브라우저를 redirect합니다. 실제 PG 연동(window.location.assign → 게이트웨이 API 호출)은 v1.0.1에서 제공될 예정입니다. 프로덕션 배포 전 릴리즈 노트를 확인하세요.

사전 조건 (순서대로 충족 필요)

  1. setAmount() 완료
  2. setOrder() 완료
  3. renderAgreement()를 통해 마운트된 약관 위젯에서 사용자가 동의 체크 (agreementChecked === true)
  4. 브라우저 환경 (typeof window !== "undefined")

파라미터

이름타입기본값설명
successUrl* string — 결제 성공 시 redirect할 URL. Query string이 자동으로 추가됩니다: paymentKey, orderId, amount.
failUrl* string — 결제 실패 시 redirect할 URL. v1 mock에서는 사용되지 않지만 필수 파라미터입니다.
customerEmail string — 고객 이메일. v1에서는 세션에 저장되지만 redirect URL에 포함되지 않습니다.

반환: Promise<void>. 성공 시 window.location.assign(successUrl + queryString)을 호출하므로 페이지가 이동합니다. Promise resolve 이후 코드는 실행되지 않습니다.

에러#

OpenCheckout SDK는 두 가지 에러 클래스를 사용합니다. 모두 Error를 extends하며 try/catch로 처리합니다.

OpenCheckoutValidationError#

필수 파라미터 누락, 잘못된 형식, 호출 순서 위반, 범위 초과 값 등 입력 유효성 문제 시 throw됩니다.

메시지 패턴발생 위치해결 방법
publishableKey is required load() publishableKey 파라미터를 전달하세요.
publishableKey must match pk_(test|live)_... load() 키 형식을 확인하세요. 테스트: pk_test_<shard>_<random>.
customerKey must be a string widgets() 비회원은 "ANONYMOUS"를 사용하세요.
customerKey must be 2-50 characters widgets() 2자 이상 50자 이하로 설정하세요.
customerKey must contain at least one of -_=.@ widgets() 특수문자(-_=.@) 최소 1개 포함 필수. "ANONYMOUS"는 예외.
locale must be one of ko, en, ja, zh-CN widgets() 지원 locale 중 하나를 사용하세요.
amount must be { value, currency } setAmount() 객체 형식으로 전달하세요.
amount.value must be a non-negative finite number setAmount() 0 이상의 유한 숫자를 사용하세요.
amount.currency must be one of KRW, USD, JPY setAmount() 지원 통화 3가지 중 하나를 사용하세요.
order.id must be a non-empty string setOrder() 비어있지 않은 주문 ID를 전달하세요.
order.name must be a non-empty string setOrder() 비어있지 않은 상품명을 전달하세요.
setAmount must be called before renderXxx renderAddress/Shipping/Payment/Agreement() setAmount()를 먼저 호출하세요.
setOrder must be called before requestPayment requestPayment() setOrder()를 먼저 호출하세요.
agreement not accepted requestPayment() 약관 위젯에서 사용자가 동의 체크를 해야 합니다.
successUrl is required requestPayment() 비어있지 않은 successUrl을 전달하세요.
failUrl is required requestPayment() 비어있지 않은 failUrl을 전달하세요.
requestPayment requires a window requestPayment() SSR 환경에서는 typeof window !== "undefined" 가드 후 호출하세요.
gatewayUrl is not a valid URL load() 유효한 http/https URL을 사용하세요.

OpenCheckoutSecurityError#

PAN(카드번호)이 위젯 경계를 넘어가려고 할 때 throw됩니다. 이 에러는 SDK 내부 보안 계층(assertPanFree)에서 자동으로 감지합니다.

메시지발생 조건해결 방법
PAN-like card data cannot cross widget boundaries setAmount(), setOrder(), renderXxx(), requestPayment() 파라미터에 Luhn 검사를 통과하는 13–19자리 숫자열이 포함될 때 카드번호는 절대 SDK 파라미터로 전달하지 마세요. 카드 입력은 게이트웨이가 제공하는 Shadow DOM 내부에서만 처리됩니다.
!

OpenCheckoutSecurityError가 예기치 않게 발생하면: 주문 ID, 상품명, 주소 필드 등에 카드번호와 유사한 숫자열(13–19자리, 구분자 포함)이 포함되어 있지 않은지 확인하세요. Luhn 알고리즘을 통과하는 숫자열은 모두 차단됩니다.

11 — Security

PAN 경계#

OpenCheckout은 카드번호(PAN)가 위젯 경계 밖으로 절대 이동할 수 없도록 설계되었습니다. 이 원칙은 ADR-003에 공식화되어 있습니다.

i

ADR-003 — PAN 경계: 카드 데이터 입력은 게이트웨이가 제공하는 별도 origin의 iframe(Shadow DOM) 내에서만 이루어집니다. SDK 파라미터, 이벤트 페이로드, 세션 상태 어디에도 카드번호가 존재할 수 없습니다. 전체 설계는 ADR-003을 참고하세요.

SDK가 하는 일#

모든 공개 API 진입점(setAmount, setOrder, renderXxx, requestPayment)은 파라미터를 assertPanFree()로 검사합니다. 이 함수는 다음 로직으로 동작합니다:

  1. 값을 재귀적으로 순회합니다 (객체, 배열, 숫자, 문자열).
  2. 13–19자리 숫자열(구분자 - · space 포함 패턴)을 추출합니다.
  3. 각 후보에 대해 반복 자릿수 필터 + Luhn 검사를 실행합니다.
  4. 검사를 통과하는 숫자열이 하나라도 있으면 OpenCheckoutSecurityError를 throw합니다.

머천트가 하면 안 되는 것#

!

다음 행동은 절대 금지입니다. SDK 자체도 이를 차단하지만, 보안 감사에서 지적될 수 있습니다:

금지 행동이유
카드번호를 order.id, order.name, URL, 이메일 등 SDK 파라미터에 포함 OpenCheckoutSecurityError가 throw되어 결제 흐름이 중단됩니다.
위젯 Shadow DOM 내부 DOM을 직접 쿼리하여 입력 값 추출 Shadow DOM 격리로 인해 기술적으로 불가능하지만, 시도 자체가 보안 정책 위반입니다.
카드번호를 localStorage, sessionStorage, 쿠키에 저장 PCI DSS 위반. SDK는 이를 방지할 수 없으므로 머천트 책임입니다.
카드번호를 머천트 서버로 직접 전송 PCI DSS 위반. 결제 확인은 paymentKey를 통해 게이트웨이 서버에서 수행하세요.

FAQ · Migration · 브라우저 지원 · 번들 크기#

자주 묻는 질문#

React/Vue/Svelte에서 사용하려면?

동일한 Vanilla JS API를 그대로 사용합니다. React의 경우 useEffect에서 load()를 호출하고, cleanup 함수에서 widgets.destroy()를 호출하세요. StrictMode에서는 effect가 두 번 실행되므로 destroyed 플래그로 guard해야 합니다 (Quick start React 탭 참고).

Next.js App Router / SSR에서 사용하려면?

SDK는 window를 필요로 하므로 SSR 단계에서는 호출하면 안 됩니다. "use client" 컴포넌트 내 useEffect에서 호출하거나, dynamic(() => import('./CheckoutWidget'), { ssr: false })로 지연 import하세요.

동일 페이지에서 복수 위젯 인스턴스를 만들 수 있나요?

네. 별도 세션이 필요한 경우 oc.widgets()를 여러 번 호출하면 됩니다. 단, 같은 체크아웃 흐름의 위젯끼리는 반드시 같은 widgets 인스턴스를 공유해야 이벤트 버스가 연결됩니다.

주소 위젯 없이 결제만 사용할 수 있나요?

네. renderPayment()와 renderAgreement()만 호출해도 됩니다. 단, renderPayment 옵션에 methods: ["card"]를 명시하지 않으면 주소 위젯에서 국가가 설정되기 전까지는 기본 KR 수단 세트가 표시됩니다.

16개국 외 국가(예: 브라질, 멕시코)는 지원하나요?

SDK에 내장된 schema는 16개국입니다. 그 외 국가는 loadCountrySchema(code)로 런타임 lazy fetch가 가능합니다. 네트워크 실패 시 FALLBACK_COUNTRY schema로 폴백됩니다.

TypeScript 타입 정의는 어디서 확인하나요?

패키지에 .d.ts가 포함되어 있습니다. 주요 타입: LoadOptions, WidgetsOptions, CheckoutWidgets, AddressSelection, ShippingSelection, Money, OrderInfo, OpenCheckoutValidationError, OpenCheckoutSecurityError.
